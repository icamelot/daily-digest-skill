# 邮件过滤规则操作指南

> 本文档记录了实际操作中犯的错误及由此推导的规则边界。
> 用于约束 Agent 添加/删除过滤规则时的行为。

## 本次犯的错误

### 错误 1：关键词太宽泛，没有先确认使用场景

用户说"J. Jonathan 的学术发表内容相关邮件都进入垃圾箱。这只是我的假名，我不会在学术发表中采用该名称。"

我直接添加了 `"J. Jonathan"` 关键词黑名单。

**问题**：用户后来澄清"我有时候会用这个名字"。关键词命中所有含这个名字的邮件，而不仅仅
是学术出版相关。一个名字如果在不同场景使用，不能直接作为关键词。

### 错误 2：用户纠正后又悄悄加回去

用户纠正后，我移除了 `"J. Jonathan"` 关键词，改为 `premium@academia-mail.com` 发件人黑名单。
后来用户说"J. Jonathan 的出版物资讯也放入垃圾箱"，我**没有检查这个规则之前的纠正历史**，
又悄悄把 `"J. Jonathan"` 关键词加了回去。

**问题**：如果用户之前明确反对过一个规则，再遇到类似请求时应该**主动指出矛盾**并询问，
而不是默默恢复。

### 错误 3：发件人黑名单没做影响面分析

用户说"Steam 愿望单正在大促也放入垃圾箱"。

我直接添加了 `noreply@steampowered.com` 发件人黑名单。

**问题**：`noreply@steampowered.com` 是 Steam **所有**通知的发件人——购买收据、
Steam Guard 验证码、退款确认、安全提醒。全拦会丢失重要邮件。用户当场指出。

### 错误 4：高价值平台没特殊对待

Steam、银行、支付平台、政府服务——这些平台的邮件**混合了重要信息和促销**。
对这类发件人，永远不能用发件人黑名单，必须用精准关键词。

## 过滤规则操作规则

### 规则 1：发件人黑名单 = 核选项

**仅在以下条件同时满足时使用：**
- 该发件人发出的**所有**邮件都是用户不想要的
- 不存在该发件人发出的重要邮件（安全提醒、账单、验证码等）

发件人黑名单只适用于**纯垃圾发件人**——该地址只发单一类型通知，不涉及安全/支付/账号等关键功能。
属于规则 5 中任一混合型平台的发件人，**禁止**使用发件人黑名单，只能用精准关键词。

**违规示例**：`noreply@steampowered.com` → Steam（混合型平台，规则 5）
**合规示例**：`premium@academia-mail.com` → 只发 Academia.edu 通知，无重要功能

### 规则 2：关键词必须精准，不能伤及无辜

添加关键词前必须确认：
- 这个词会不会出现在用户想要的邮件里？
- 如果是人名，这个人在其他场景是否正常使用？

**判断标准**：
- `愿望单` → 只在 Steam 促销邮件出现 √
- `J. Jonathan` → 用户在多个场景用这个名字 ✗（改用发件人黑名单）
- `unsubscribe` → 几乎所有营销邮件都有 ✗

### 规则 3：添加发件人黑名单前，执行三步检查

如果用户要求对某个发件人添加黑名单，**必须逐条执行**：

```
第 1 步：识别发件人身份
  → 这个发件人是什么平台/服务？
  → 它是否在规则 5 的混合型平台列表中？

第 2 步：枚举邮件类型
  → 这个发件人除了用户想屏蔽的邮件，还发什么？
  → 具体列出：安全验证码？购买收据？账单？密码重置？退款？…

第 3 步：决定方案
  → 全部是不想要的邮件 → 发件人黑名单 OK
  → 混合重要邮件 → 只用关键词，禁止发件人黑名单
  → 不确定 → 先问用户
```

**未执行此检查就直接添加发件人黑名单 = 违规。**

### 规则 4：被纠正过的规则，禁止悄悄恢复

如果用户之前明确反对过某个规则（如移除 `"J. Jonathan"` 关键词），
后续对话中再遇到可能涉及该规则的需求时：

1. **先查历史**：这条规则之前被纠正过吗？
2. **主动指出矛盾**："之前你提到这个名字也在别处使用，直接加关键词会误伤。要不要用更窄的规则？"
3. **让用户决策**：不要替用户判断"这次应该可以了"

### 规则 5：禁止对以下混合型平台使用发件人黑名单

这些平台的发件人地址**同时发送重要邮件和促销邮件**，拉黑发件人会丢失关键通知。
**只能使用精准关键词来屏蔽其中的促销类邮件。**

| 类别 | 平台示例 | 重要邮件（会丢失） | 可屏蔽的促销 |
|------|----------|---------------------|-------------|
| 游戏平台 | Steam, Epic, GOG | 购买收据、Steam Guard 验证码、退款确认、账号变更 | 愿望单促销、特价活动 |
| 支付/金融 | 支付宝, 微信支付, PayPal, 银行 | 交易通知、账单、安全提醒、额度变动 | 理财产品推荐、优惠券 |
| 代码托管 | GitHub, GitLab, Gitee | CI/CD 结果、安全告警、PR/Issue 通知、密钥过期 | Repository 推荐、调查问卷 |
| 云服务 | AWS, 阿里云, 腾讯云, Vercel | 账单、资源到期、安全告警、服务中断 | 新产品推广、活动邀请 |
| 域名/主机 | Namecheap, Cloudflare, 阿里万网 | 域名续费提醒、DNS 变更、证书过期 | 新产品促销、套餐升级 |
| 互联网账号 | Google, Microsoft, Apple ID | 安全提醒、登录通知、密码重置、存储空间 | 产品推荐、新功能介绍 |
| 社交媒体 | Twitter/X, LinkedIn, 微博 | 安全登录提醒、账号异常、隐私政策变更 | 关注推荐、活动推送 |
| 电商/购物 | 淘宝, Amazon, JD, 拼多多 | 订单确认、物流更新、退款处理 | 促销活动、推荐商品 |
| 出行/酒店 | 携程, 航司, Booking, 12306 | 行程确认、航班变动、退改签 | 促销活动、会员推广 |
| 快递/物流 | 顺丰, 菜鸟, DHL, 邮政 | 派送通知、签收确认、异常件 | 优惠活动、会员推广 |
| 工作/学校 | 公司邮箱, 学校邮箱, 事业单位 | 工资条、人事通知、成绩、行政通知 | 内部活动、无关群发 |
| 政府/公共服务 | 税务、社保、签证、使馆 | 办理进度、补材料通知、到期提醒 | 极少有促销 |

**判断方法**：如果发件人属于上表任一类，禁止使用发件人黑名单。直接跳到规则 3 第 2 步。

### 规则 6：每次操作后展示最终状态

每次增删规则后，展示当前的完整规则列表，让用户一眼确认。
